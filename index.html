Filing Issues

Issue trackers are awesome, but there are a few tips filers should keep in mind:

  1. Use a short descriptive title. Many good issues begin with a verb.
  2. Know the difference between a bug and feature
  3. For bugs include a 'Steps to Replicate:' section
  4. A 'Prefered Behaviour:' section can be included if appropriate
  5. Don't feel too bad if issues aren't accepted. Some ideas aren't practical in the short term, but may come back.
  6. All discussion should add to the resolution. No 'me too!'s

Read the GitHub guide about issues for more information.

Submitting Patches

There are a few rules to follow:

  1. No patches without an issue (including discussion) beforehand
  2. One issue per patch (there is wiggle room for related issues)
  3. Include issue number(s) in the patch/commit comment
  4. Patch against the beta branch
  5. Run tests before submitting
  6. Custom code is gifted to the project (Attribution for significant contributions may be worked out)

I suspect the last point will raise the biggest hackles, but there can be all sorts of hassles otherwise. Most notably is if sometime in the decades to come we decide to change from the MIT license to something else.
